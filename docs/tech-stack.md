# 기술 스택

본 문서는 **launchpad (DdangHa)** 프로젝트에서 사용하는  
기술 스택과 그 선택 이유를 설명한다.

---

## Backend

### FastAPI

**선택 이유**

- Python 기반으로 팀원 진입장벽이 낮음
- 타입 힌트 기반으로 스펙과 코드 연결이 자연스러움
- 비동기 지원으로 확장성 확보
- Lambda 환경과 궁합이 좋음

**왜 다른 프레임워크가 아닌가**

- Django: 과도한 기능, 빌더톤에 무거움
- Spring Boot: 초기 세팅 비용이 큼
- Express: 타입 안정성과 스펙 연결이 상대적으로 약함

---

### AWS Lambda + API Gateway

**선택 이유**

- 서버 관리 불필요
- 트래픽이 없으면 비용 0에 가까움
- 빌더톤 환경에서 배포/롤백이 빠름
- 이후 확장 시에도 구조 변경 최소화

**빌더톤 관점의 장점**

- “서버 띄우는 것”에 신경 쓸 필요 없음
- 기능 단위로 배포 가능

---

### Database (Railway)

**선택 이유**

- 이미 운영 경험 있음
- 빠른 생성과 관리
- 로컬 ↔ 클라우드 환경 일관성 유지 쉬움

**전제**

- 고가용성, 대규모 트래픽은 고려하지 않음
- 빌더톤 및 데모 목적

---

## Frontend

### React + TypeScript

**선택 이유**

- 시장 표준
- 팀원 학습 자료 풍부
- 타입 기반 협업에 유리

---

### Feature-Sliced Design (FSD)

**선택 이유**

- 기능 단위 사고를 강제
- 기획 변경에 강함
- Admin / User UI 확장에 적합

**빌더톤에서의 장점**

- 역할 분담이 명확
- 작업 충돌 최소화

---

## Infra / Tooling

### Docker (로컬 개발용)

**선택 이유**

- 로컬 환경 통일
- 팀원 간 “내 컴퓨터에서는 되는데” 방지
- Lambda 배포 전 사전 검증 가능

---

### GitHub Actions (CI)

**선택 이유**

- GitHub native
- 설정 단순
- 빌더톤 기간에 유지보수 부담 적음

---

### Monorepo (yarn workspaces)

**선택 이유**

- Backend / Frontend / Admin 통합 관리
- 공통 코드 공유 용이
- Spec, Docs를 루트에서 함께 관리 가능

---

## Payment (Test Only)

### 테스트 결제 전략

- 실제 결제 연동 없음
- PG사 테스트 모드 또는 Mock 사용
- 도메인 구조와 데이터 흐름 검증이 목적

---

## 의도적으로 선택하지 않은 것들

- Terraform / CDK
- 마이크로서비스
- 복잡한 캐시 구조
- 메시지 큐

> 빌더톤에서는 “안 쓰는 기술을 안 쓰는 것”도 중요한 선택이다.
